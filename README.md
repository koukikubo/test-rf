# RFリピート分析・予約管理アプリ

本プロジェクトは、顧客の来店履歴をもとに **来店回数（F）** と **最終来店時期（R）** を集計し、  
**RFランクの可視化** と **予約管理** を行うアプリケーションです。

Rails API と Next.js フロントエンドで構成しており、  
将来的には飲食店向け CRM（顧客関係管理）へ拡張することを見据えて開発しています。

---

## 概要

本アプリは、飲食店における

- 予約管理
- 顧客管理
- リピート分析

を一体で扱うための土台づくりを目的としています。

単純な顧客台帳や予約台帳ではなく、  
「最近来店しているか」「何回来店しているか」をもとに顧客状態を把握し、  
**RFランク** や **RF分析表** として可視化できる構成を目指しています。

### 実装済みの主な機能

- **顧客管理**
  - 登録
  - 一覧表示
  - 詳細表示
- **予約管理**
  - 登録
  - 一覧表示
  - 詳細表示
  - 更新
  - 削除
- **RFランク集計**
  - 顧客ごとの来店回数・最終来店日の集計
  - RFスコア保存
  - ランキング表示
- **RF分析表**
  - 来店期間 × 来店回数帯で可視化
  - セル別の対象顧客一覧表示
- **RFマスタ管理**
  - ランク条件の一覧表示
  - 編集
- **非同期処理**
  - 予約操作時の RF スコア更新 Job 実行
- **定期実行**
  - `whenever` を利用した RF スコア全体更新設定
- **テスト**
  - Request Spec / Service Spec による主要ロジックの検証

## 目的・背景

### 目的

- 顧客ごとの来店状況を可視化し、リピート分析を可能にする
- 予約・顧客・RF情報を一元管理する基盤を構築する
- 将来的な飲食店向け CRM 機能拡張に耐えられるデータ構造を整える

### 背景

一般的な顧客管理では「情報を登録する」ことが中心になりやすく、  
「いつ来たのか」「何回来ているのか」「どの顧客を優先して把握すべきか」といった分析につながりにくいケースがあります。

本アプリでは、予約履歴から顧客状態を自動集計し、  
RFランクや分析表を通じて **現場で使える見え方** を意識した設計を進めています。

## RFランクの考え方

本アプリでは、以下の2指標を利用して顧客状態を把握します。

- **R（Recency）**: 最終来店時期
- **F（Frequency）**: 来店回数

### 現在のランク区分

現在は、次のランクを使用しています。

- **A** : 超常連顧客
- **B** : 常連客
- **C** : A / B / D / E 以外
- **D** : 休眠客
- **E** : 新規顧客
- **Z** : ランク外
- **対象外** : 集計対象期間外

### 補足

ランク条件は今後も見直しや拡張を想定しています。  
現時点では、期間条件は固定寄りで扱い、  
主に来店回数条件や表示ロジックを中心に調整を進めています。

## RFスコア更新の仕組み

### 1. 個別更新（予約操作時）

予約の **作成 / 更新 / 削除** 成功時に、  
対象顧客の RF スコア更新 Job を実行します。

現在は学習・挙動確認を優先し、  
予約操作成功後に Controller から `RfScoreUpdateJob` を enqueue する構成にしています。

### 2. 全体更新（定期実行）

予約操作が発生しない顧客についても、時間経過によるランク変化を反映するため、  
全顧客を対象とした RF スコア一括更新タスクを用意しています。

docker compose exec backend bin/rails rf:update_rf_scores
